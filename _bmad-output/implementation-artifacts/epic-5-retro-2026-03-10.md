## What Went Well

- Container baseline and orchestration became reproducible and reviewer-friendly across default and profiled Compose paths.
- QA gates were tightened with explicit coverage enforcement and maintained green status while scope expanded.
- Evidence quality improved significantly: QA, performance, and security now have dated, navigable artifacts.

- Story flow remained incremental and traceable, with clear linkage from acceptance criteria to executed commands.

## Challenges And Friction

- Multi-stage Docker target ambiguity caused a real runtime regression risk (`SEC-001`) until Compose explicitly pinned `target: runtime`.
- Test profile and runtime assumptions required iterative hardening to avoid drift (for example, backend test stage and permissions alignment).
- Coverage-instrumented warnings introduced noise and required fixture/lifecycle hygiene work plus clear risk framing.
- Accessibility evidence scope was initially overstated and needed correction to match actual asserted rules.
- Several docs required repeated synchronization after infrastructure adjustments to keep reviewer confidence high.

## Story Pattern Synthesis

Common struggles (appeared in multiple stories):
- Infrastructure correctness issues were often revealed only under realistic runtime execution (not static config inspection alone).
- Documentation and evidence lagged behind implementation until explicitly treated as first-class deliverables.
- Review findings clustered around traceability, environment parity, and hidden default behavior in tooling.

Recurring review feedback:
- Require explicit runtime intent in Docker/Compose.
- Keep environment strategy DRY and profile-aware.
- Avoid over-claiming validation scope; reflect exactly what was asserted.
- Cross-link artifacts so reviewers can verify claims quickly.

Breakthrough moments:
- Introducing dated evidence artifacts (`security-review`, `qa-tooling-evidence`, `performance-audit`) made compliance verification fast.
- Compose profile strategy (`dev`, `test`) improved reproducibility while preserving simple default startup.

## Key Lessons

1. Default behavior in tooling can introduce hidden runtime risk unless intent is explicit (`target`, profiles, env wiring).
2. Evidence discoverability is not optional for production-readiness; it must be built into story completion.
3. Fast review turnarounds are most effective when changes are scoped and mapped directly to finding IDs.
4. Quality claims must align exactly with what automation proves.
